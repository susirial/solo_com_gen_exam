# Exam Paper Generator

一个用于中文 K-12 考卷生成的工作流型 Skill。这个 Skill 必须先确认用户需求，再生成组卷方案；只有在方案已明确或用户已确认后，才生成完整试卷内容。需要输出页面时，再调用内置脚本生成教师版或学生版 HTML。

## 何时使用

在以下场景优先使用本 Skill：

- 用户要生成试卷、考卷、练习卷、测试卷、月考试卷、单元卷
- 用户要按年级、学科、知识点、题型、难度来命题或组卷
- 用户要重生某题、换题、补解析、调难度、整理教师版/学生版
- 用户要把一套卷子整理成结构化内容或 HTML 页面

在以下场景不要使用本 Skill：

- 用户只是问学科知识点，不涉及试卷或题目生成
- 用户只是调试 Python 代码、模板代码或脚本实现
- 用户要处理与试卷无关的通用文档排版任务

## 工作原则

- 先确认，不要直接出题
- 先方案，后内容
- 优先结构化输出，再衍生 HTML
- 缺少关键信息时主动追问，不要臆造
- 不假设外部 API、数据库、Web 服务或本地项目代码可用

## 先做需求确认

开始执行时，先判断用户请求属于哪一类：

1. 新建整卷
2. 调整现有组卷方案
3. 重生单题或编辑题目
4. 把已有内容整理成教师版 / 学生版 HTML

随后按需确认以下输入。若缺失关键项，必须先追问：

- 学段、年级、学科
- 试卷类型与考试场景
- 总分、时长
- 分节题型、题量、单题分值
- 知识点范围
- 难度要求
- 是否需要答案、解析
- 是否需要教师版 HTML、学生版 HTML 或两者都要

详细确认清单见 `references/input-checklist.md`。

## 标准工作流

### 阶段 1：确认需求

- 识别任务类型
- 收集最小必要输入
- 若用户提供的信息不完整，先问清再继续

### 阶段 2：生成组卷方案

在没有明确方案前，不要直接生成完整试卷。先输出一份组卷方案，至少包含：

- 试卷标题建议
- 分节结构
- 每节题型、题量、分值
- 难度分布
- 知识点覆盖
- 风险与待确认项

若需要校验方案结构，可把方案整理为 JSON 后调用：

```bash
python3 scripts/validate_plan.py path/to/plan.json
```

### 阶段 3：等待用户确认方案

方案输出后，明确提示用户确认。只有在以下任一条件成立时，才能进入内容生成：

- 用户明确说“确认方案”“按此生成”“开始出题”
- 用户直接给出已完整可执行的方案并要求继续

### 阶段 4：生成完整试卷内容

生成内容时，优先整理为统一结构。字段约定见 `references/schema.md`。

内容至少应包含：

- 试卷标题与元信息
- section 列表
- 每题题干
- 选项（如适用）
- 标准答案
- 解析或评分要点
- 分值、难度、知识点

若内容需要统一字段、补默认值或重算总分，可调用：

```bash
python3 scripts/normalize_paper.py path/to/paper.json -o path/to/paper.normalized.json
```

若需要校验完整试卷结构，可调用：

```bash
python3 scripts/validate_paper.py path/to/paper.normalized.json
```

### 阶段 5：生成 HTML

当用户要求教师版 / 学生版 HTML 时，先确保试卷内容已整理成 `references/schema.md` 中定义的结构，再调用：

```bash
python3 scripts/render_html.py path/to/paper.normalized.json --variant teacher -o teacher.html
python3 scripts/render_html.py path/to/paper.normalized.json --variant student -o student.html
```

渲染规则见 `references/rendering.md`。

## 输出要求

### 方案阶段输出

始终优先输出：

- 已确认输入
- 建议方案
- 待确认项

若用户信息不足，不要伪造完整方案。

### 内容阶段输出

优先输出为结构化内容，建议使用清晰的 JSON 或分节 Markdown。若用户明确要求 HTML，再继续渲染页面。

### HTML 输出

- 教师版显示答案、解析、评分要点、知识点
- 学生版不显示答案和解析
- 两个版本必须来源于同一份统一结构，不维护两套内容源

## 分支处理

### 重生单题

若用户要重生某题：

- 先确认题号、所属分节、目标难度、避让知识点、教师备注
- 明确这是“重生”而不是“编辑”
- 只替换目标题，除非用户要求整节或整卷重做

### 编辑题目

若用户要编辑题目：

- 先确认动作是改写题干、补解析、调难度还是翻译
- 尽量保持原题考查点不变，除非用户另有要求

### 仅渲染 HTML

若用户已经提供完整结构化内容：

- 不重复生成方案或题目
- 直接检查结构合法性
- 合法后渲染教师版 / 学生版 HTML

## 边界与风险

- 不引用本地其他目录中的 Python 代码
- 不假设存在外部模型、数据库或 Web 服务
- 不把缺失数据模型写成确定无疑的真实契约
- 无法确认的信息要显式标出，而不是自行补全为“看起来合理”的内容

详细限制见 `references/boundaries.md`。

## 示例

### 示例 1：整卷生成

输入：

> 给初二数学生成一张 100 分钟期中试卷，分选择、填空、解答三节，先给我组卷方案，确认后再出完整试卷，并生成教师版和学生版 HTML。

正确行为：

- 先确认题量、分值、知识点与难度
- 先输出方案
- 等待确认
- 再生成完整试卷内容
- 最后渲染 HTML

### 示例 2：重生单题

输入：

> 把阅读理解第 2 题重生一版，难度提高，但不要重复原知识点。

正确行为：

- 先确认所属 section、原题约束和目标难度
- 只处理目标题
- 不自动改整卷

### 示例 3：只做 HTML

输入：

> 我已经有结构化试卷 JSON，帮我生成教师版和学生版 HTML。

正确行为：

- 先检查结构是否合法
- 合法后直接渲染 HTML
- 不重新生成题目
